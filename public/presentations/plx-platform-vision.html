The moment we're in

Agents do the work. Humans direct the work.

We have three active engineering surfaces — Mission Control, the Customer Portal, and shared Infra — that must share one brand, one governance model, and one traceable record. The opportunity is speed with accountability, not speed without it.

  • Everything resolves to a Task — PRD → task → PR → evidence → merge. No orphan work.
  • SharePoint is the system of record — the app is a fast lens; conflicts are resolved by a human, never auto-merged.
  • One definition of "green" — ./scripts/preflight.sh locally and in CI; no bypassing the gate.

Customer Portal05 / 11
PLX_Infra

Infra — the floor we all stand on

Shared infrastructure turns agent speed into team-safe speed: one secrets path, one CI definition, one staging environment, one compliance story across repos.

  • Secrets — AWS Secrets Manager via one accessor; no scattered env lookups or committed keys.
  • Gates — preflight.sh pre-commit / pre-push / CI; GitHub App compliance checkout stamps PRs with MC-Checkout.
  • Staging — plx-postgres-staging, Vercel staging URL, ~/.secrets-env.staging only.
  • Integrations — MCP servers ship disabled; operator enables per session with auditable kill switches.

Project Orchestrator — a skill, not an agent

Multi-file features, migrations, and cross-repo efforts get a written SPEC.md with a phase DAG. A human approves once; an agent invokes this skill to execute phases in isolated worktrees and integrate behind one PR.

Project Orchestrator skill10 / 14
The contract

Every project gets a SPEC you can read in ten minutes

Phase DAG

Ordered dependencies

Phases declare depends_on, owns globs, and forbidden paths. Scope-check rejects drift before merge.

Acceptance

Exit-0 proof

Each phase names a command — usually preflight.sh or targeted tests. "Done" means the command passed, not a chat claim.

Evidence

Dated bundles

Reports land in artifacts/<domain>/<date>-<slug>/ with REPORT.md — never loose files at repo root.

Delivery

One PR theme

Integration branch proj/<slug>/integration → single PR with rollback plan and linked MC tasks.

Before the next meeting

Design a skill your team needs

Think about a repeatable workflow in your domain — something you'd run weekly that an agent could follow with human gates. Not an agent persona; a playbook.

Your assignment

  1. Pick a pain point — onboarding checklists, UAT triage, formula review prep, sync conflict runbook, anything you repeat.
  2. Draft a skill spec in Cursor or Claude: name, trigger ("when to use"), inputs, step-by-step workflow, success criteria, human gates.
  3. Map it to the stack — which MCP tools, which repo (MC / Portal / Infra), which context layer it reads.
  4. Bring the spec to the next meeting — we'll review, combine, and prioritize 1–2 to build.

Template hint: see .cursor/skills/ or SKILL.md format · questions welcome now
